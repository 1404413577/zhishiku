# 知识库化改造计划

## 背景判断

当前项目已经具备个人笔记系统的基础能力：Markdown 编辑、文档树、标签、搜索、图谱、思维导图、AI 对话、导入导出和本地文件夹工作区。但它目前的核心对象仍然是“文档”，使用方式更接近记笔记。

真正的知识库不只是保存笔记，而是持续沉淀可复用知识。它需要回答这些问题：

- 这条内容是什么类型的知识？
- 它是否可信、是否过期、是否已经整理完成？
- 它引用了哪些知识？又被哪些知识引用？
- 用户能否从问题直接找到答案，而不是只找到文档？
- 系统能否提示哪些知识需要维护？

因此改造目标不是推倒重来，而是在现有文档系统上增加“知识结构层”。

## 目标形态

把项目从“Markdown 笔记管理器”升级为“个人知识库工作台”：

- 文档有知识类型、状态、可信度、来源和别名。
- 新建内容优先从知识模板开始，而不是空白页。
- 文档之间有双链、反向链接、未链接提及和关系类型。
- 搜索支持按知识类型、状态、标签过滤，并显示命中上下文。
- 首页从普通统计页升级为知识健康仪表盘。
- AI 对话能够基于知识库回答，并返回引用来源。

## 设计原则

1. 保留现有存储模式  
   IndexedDB、本地文件夹、预设文档都继续可用。新增字段必须兼容旧数据。

2. 先元数据，后智能化  
   先让知识有结构，再让 AI 自动摘要、打标签和推荐关联。

3. 文档仍是底层载体  
   不引入复杂数据库模型。每篇 Markdown 文档附加知识库元数据。

4. 低成本渐进迁移  
   旧文档默认归类为 `note`，状态为 `draft`，可信度为 `medium`。

5. 本地优先  
   所有知识库能力优先在前端本地完成，AI 能力作为增强层。

## 核心数据模型

建议在现有文档节点上扩展以下字段：

```ts
knowledgeType:
  | 'note'
  | 'concept'
  | 'guide'
  | 'decision'
  | 'faq'
  | 'source'
  | 'case'

knowledgeStatus:
  | 'draft'
  | 'verified'
  | 'outdated'
  | 'archived'

confidence:
  | 'low'
  | 'medium'
  | 'high'

aliases: string[]
sourceUrl: string
relatedIds: string[]
reviewedAt: string | null
```

字段含义：

- `knowledgeType`：知识类型。决定模板、筛选和后续 AI 处理方式。
- `knowledgeStatus`：整理状态。用于健康仪表盘和维护提醒。
- `confidence`：可信度。用于区分临时记录和已验证知识。
- `aliases`：别名。用于搜索、未链接提及和概念识别。
- `sourceUrl`：外部来源。用于资料摘录和引用追踪。
- `relatedIds`：人工关联文档。区别于自动解析的双链。
- `reviewedAt`：最近复核时间。用于过期提醒。

## 知识类型定义

### note

普通记录。用于兼容旧文档和日常快速输入。

### concept

概念说明。适合术语、模型、技术点、方法论。

建议结构：

```md
# 概念名称

## 一句话解释

## 详细说明

## 适用场景

## 易混淆概念

## 相关知识

## 来源
```

### guide

操作指南。适合流程、教程、SOP。

建议结构：

```md
# 指南标题

## 目标

## 前置条件

## 操作步骤

## 验证方式

## 常见问题

## 相关知识
```

### decision

决策记录。适合技术选型、方案取舍。

建议结构：

```md
# 决策标题

## 背景

## 备选方案

## 决策

## 理由

## 影响

## 后续复核
```

### faq

问答知识。适合常见问题、排障。

建议结构：

```md
# 问题

## 简短答案

## 详细解释

## 适用条件

## 相关问题
```

### source

资料摘录。适合文章、书籍、网页、论文、视频。

建议结构：

```md
# 资料标题

## 来源

## 摘要

## 关键摘录

## 我的理解

## 可转化的知识点
```

### case

经验案例。适合问题复盘、项目经验。

建议结构：

```md
# 案例标题

## 场景

## 问题

## 原因

## 解决方案

## 验证结果

## 经验沉淀
```

## 功能路线

### 第一阶段：知识结构基础

目标：让系统开始“识别知识”，而不只是保存文档。

任务：

- 扩展文档类型定义，增加知识库元数据字段。
- 文档 normalize/create 时补齐默认元数据。
- 增加知识类型模板。
- 新建文档时保留现有模板选择，但模板应覆盖知识库常见类型。
- 旧文档兼容：默认 `knowledgeType = 'note'`。

验收：

- 旧数据加载不报错。
- 新建文档有默认知识库字段。
- 模板列表包含概念、指南、决策、FAQ、资料、案例。

### 第二阶段：知识维护与健康仪表盘

目标：首页从统计页变成知识库维护入口。

任务：

- 增加知识健康指标：
  - 草稿数量
  - 已验证数量
  - 过期知识数量
  - 无标签文档数量
  - 孤立文档数量
  - 缺摘要文档数量
- 首页展示“待整理”“待复核”“孤立知识”。
- 文档列表支持按 `knowledgeType` 和 `knowledgeStatus` 过滤。

验收：

- 用户能看到哪些知识需要整理。
- 用户能快速进入低质量或待维护文档。

### 第三阶段：双链与关系面板

目标：让知识之间产生可导航关系。

任务：

- 解析 Markdown 中的 `[[文档标题]]` 双链。
- 生成正向链接和反向链接。
- 在查看页增加“相关知识”面板：
  - 引用的文档
  - 被引用的文档
  - 手动关联文档
  - 未链接提及
- 图谱节点区分知识类型。

验收：

- 打开任意文档能看到它和其他知识的关系。
- 点击反向链接能跳转到来源文档。

### 第四阶段：搜索升级

目标：搜索从“找文档”升级为“找知识”。

任务：

- 搜索结果展示命中片段。
- 支持类型、状态、标签组合过滤。
- 支持别名命中。
- 搜索结果标记知识状态和可信度。

验收：

- 搜索一个概念时，能找到标题、别名、正文命中的结果。
- 用户能筛选“已验证指南”“过期概念”等集合。

### 第五阶段：AI 知识库问答

目标：AI 对话从普通聊天升级为基于知识库的问答。

任务：

- 从搜索服务取相关文档作为上下文。
- AI 回答时要求引用来源文档。
- AI 可执行：
  - 自动摘要
  - 自动标签
  - 推荐关联
  - 从资料摘录生成概念卡片
- 对低可信内容给出提示。

验收：

- AI 回答包含来源列表。
- 用户能从回答跳转到引用文档。
- AI 不直接把低可信草稿当作事实输出。

- 今日待整理
- 最近更新
- 知识健康分
- 孤立知识
- 过期知识

- 状态
- 可信度

- 摘要

- 反向链接

## 技术实现建议

### 元数据兼容

所有新增字段必须在 normalize 层补默认值，避免旧数据报错。

### 关系计算

双链、反链、孤立文档优先做成纯函数服务，不直接写进文档：

```ts
buildKnowledgeRelations(documents)
getBacklinks(documentId, relations)
getUnlinkedMentions(document, documents)
```

### 健康分计算

健康分不需要存储，按当前文档集合计算：

```ts
health = {
  total,
  drafts,
  outdated,
  missingTags,
  missingSummary,
  isolated,
  score
}
```

### AI 上下文

AI 问答的上下文应由搜索结果构建，不直接塞入全部文档：

```ts
retrieveRelevantDocuments(question, filters)
buildKnowledgeContext(results)
askWithCitations(question, context)
```
