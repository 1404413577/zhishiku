# 模块边界与依赖规则

## 1. 文档目的

本文档用于约束 V2 重构后的模块边界，防止项目再次变成高耦合、低内聚结构。

核心目标：

- 页面不直接碰底层存储。
- Store 不承载复杂业务。
- 服务层统一编排业务流程。
- 领域层保持纯净。
- 依赖方向单向、可理解、可测试。

## 2. 目标分层

```text
app
  ↓
features
  ↓
stores
  ↓
services
  ↓
repositories
  ↓
browser/api

domain 可被 stores/services/repositories/features 引用，但不能反向依赖它们。
shared 可被多数模块引用，但不能依赖具体业务模块。
```

## 3. 模块职责

| 模块 | 允许做什么 | 禁止做什么 |
| --- | --- | --- |
| `app` | 应用启动、路由、全局布局注册 | 写具体业务逻辑 |
| `features` | 页面、业务组件、用户交互 | 直接访问 IndexedDB/WebDAV/FileSystem |
| `stores` | 响应式状态、当前选择、加载状态 | 实现复杂业务和底层读写 |
| `services` | 编排业务流程、调用 repository、调用 domain 规则 | 直接操作 DOM 或依赖 Vue 组件 |
| `repositories` | 封装数据读写 | 引用 Vue、Element Plus、页面组件 |
| `domain` | 类型、纯函数、业务规则 | 发请求、读写浏览器存储 |
| `shared` | 通用组件、工具、hooks、样式 | 依赖具体 feature |
| `workers` | 后台计算、索引、关系分析 | 直接修改 Pinia 状态 |

## 4. 依赖规则

### 4.1 允许依赖

- `features` 可以依赖 `stores`、`services`、`shared`、`domain`。
- `stores` 可以依赖 `services`、`domain`。
- `services` 可以依赖 `repositories`、`domain`、`shared/utils`。
- `repositories` 可以依赖 `domain`、浏览器 API、第三方存储库。
- `shared` 可以依赖通用第三方库。
- `domain` 只能依赖 TypeScript 标准能力和纯工具。

### 4.2 禁止依赖

- `domain` 禁止依赖 `services`、`stores`、`features`。
- `repositories` 禁止依赖 `stores`、`features`、Vue 组件。
- `services` 禁止依赖 Vue 页面组件。
- `shared` 禁止依赖具体业务模块。
- 页面禁止直接依赖 `localforage`、`fetch WebDAV`、`FileSystemHandle`。
- 页面禁止直接创建搜索索引。

## 5. 页面组件规则

页面组件应该：

- 调用 Store 获取状态。
- 调用 Service 执行业务动作。
- 管理局部 UI 状态。
- 处理用户事件。
- 展示加载、错误、空状态。

页面组件不应该：

- 直接读写 IndexedDB。
- 直接拼接 WebDAV 请求。
- 直接维护搜索索引。
- 直接实现数据迁移。
- 包含超过一个模块的复杂业务流程。

## 6. Store 规则

Store 应该保存：

- 当前工作区。
- 文档列表缓存。
- 当前文档 id。
- 加载状态。
- 错误状态。
- 当前筛选条件。

Store 不应该：

- 直接访问底层存储库。
- 写复杂树形移动规则。
- 写 Markdown 解析逻辑。
- 写 AI Provider 调用细节。
- 写 WebDAV 同步流程。

推荐模式：

```ts
export const useDocumentStore = defineStore('document', () => {
  const nodes = ref<KnowledgeNode[]>([])
  const loading = ref(false)

  async function loadDocuments() {
    loading.value = true
    try {
      nodes.value = await documentService.list()
    } finally {
      loading.value = false
    }
  }

  return { nodes, loading, loadDocuments }
})
```

## 7. Service 规则

Service 应该：

- 作为业务用例入口。
- 组合多个 repository。
- 执行领域规则校验。
- 处理错误归一化。
- 发出业务事件，例如文档更新后通知搜索索引。

Service 不应该：

- 管理复杂组件状态。
- 直接显示 Element Plus Message。
- 依赖某个页面。

错误应返回或抛出标准错误，由页面决定如何提示。

## 8. Repository 规则

Repository 应该：

- 只关注数据读写。
- 隐藏 IndexedDB、本地文件夹、WebDAV 等实现差异。
- 输入输出使用领域模型。
- 不关心 UI。

Repository 不应该：

- 弹窗提示。
- 路由跳转。
- 修改 Pinia。
- 调用 AI。

## 9. Shared 规则

`shared` 适合放：

- 通用按钮、空状态、错误状态组件。
- `useAsyncTask`、`useConfirm` 等通用 composable。
- 日期格式化、字符串处理。
- 常量。
- 全局样式。

`shared` 不适合放：

- 文档业务逻辑。
- AI 对话逻辑。
- 思维导图节点操作。
- 只被一个页面使用的组件。

## 10. Feature 目录规则

每个 feature 目录建议结构：

```text
features/editor/
├── pages/
├── components/
├── composables/
├── services/
├── types.ts
└── index.ts
```

如果某个 service 只服务该 feature，可以先放在 feature 内。若被多个 feature 复用，再上移到全局 `services`。

## 11. 命名规则

- 领域类型：`KnowledgeDocument`、`Workspace`。
- Service：`documentService`、`searchService`。
- Repository：`indexedDbDocumentRepository`。
- Store：`useDocumentStore`。
- Composable：`useAutoSave`、`useEditorDocument`。
- 页面：`EditorPage.vue`。
- 组件：`EditorToolbar.vue`。

## 12. 禁止事项

- 禁止在页面中直接 `localStorage.setItem` 保存业务数据。
- 禁止在多个页面复制 Markdown 渲染配置。
- 禁止在 Store 中直接 new AI Provider。
- 禁止在 Service 中 import Vue 页面组件。
- 禁止跨 feature 深层引用内部文件，例如 `features/editor/components/...`。
- 禁止新增 `.bak` 代码文件。
- 禁止为了临时修复绕过 Service 直接改 repository。

## 13. 判断标准

一个功能放在哪里，可以用以下问题判断：

- 是否是纯规则？放 `domain`。
- 是否是数据读写？放 `repositories`。
- 是否是业务流程？放 `services`。
- 是否是响应式状态？放 `stores`。
- 是否是页面交互？放 `features`。
- 是否多个模块通用且无业务依赖？放 `shared`。
