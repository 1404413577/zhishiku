# 知识库项目重构方案

## 1. 背景

当前项目是围绕多个想法逐步叠加完成的，功能已经比较丰富，但架构上存在明显的高耦合、低内聚问题。典型表现包括：

- 页面组件直接承担大量业务逻辑。
- Store 同时负责状态、业务编排、存储调用和搜索索引更新。
- IndexedDB、本地文件夹、WebDAV 等存储能力混杂在文档逻辑中。
- Markdown 渲染、编辑器、阅读器、静态文档浏览存在重复实现。
- AI 在线接口、本地模型、Ollama 的能力边界不够清晰。
- 搜索、图谱、统计等派生能力直接依赖原始文档列表，缺少统一关系层。
- 复杂功能集中在单文件中，维护成本高，后续新增功能容易继续堆叠。

重构目标不是简单换目录，而是重新梳理业务边界，让项目从“功能堆叠型应用”升级为“模块化知识库系统”。

## 2. 重构目标

### 2.1 架构目标

- 建立清晰分层：领域模型、数据访问、业务服务、状态管理、功能模块、UI 组件。
- 页面只负责交互和展示，不直接处理底层存储、搜索索引、AI 请求和同步细节。
- Store 只保存状态和调用服务，不承载复杂业务规则。
- 所有核心能力通过服务层统一暴露。
- 为后续扩展预留稳定接口，例如更多 AI Provider、更多同步方式、更多导出格式。

### 2.2 体验目标

- 提升文档编辑、保存、搜索、阅读的稳定性。
- 所有长任务都有明确状态：加载中、成功、失败、可取消。
- 本地存储和同步失败等状态对用户可见、可恢复；不再启用 PWA 离线缓存。
- 保持核心路径简洁：打开应用、查找文档、编辑保存、阅读跳转。
- 减少页面卡顿，搜索和索引任务尽量放入 Worker。

### 2.3 代码目标

- 使用 TypeScript 逐步覆盖核心模型和服务层。
- 核心数据结构显式定义，避免临时字段四处分散。
- 建立小而稳定的模块接口。
- 添加必要测试，优先覆盖数据转换、存储、搜索、Markdown 解析等高风险逻辑。
- 删除重复实现和废弃代码。

## 3. 推荐总体架构

```text
src/
├── app/                    # 应用启动、路由、布局、全局插件
├── domain/                 # 领域模型和纯业务规则
├── repositories/           # 数据读写实现
├── services/               # 业务服务编排
├── stores/                 # Pinia 状态
├── features/               # 按业务功能划分页面和组件
├── shared/                 # 通用组件、hooks、工具函数、常量
└── workers/                # 搜索、索引、关系计算等后台任务
```

### 3.1 分层职责

| 层级 | 职责 | 不应该做的事 |
| --- | --- | --- |
| `domain` | 定义核心类型、纯函数、业务规则 | 调用浏览器 API、发请求、改 UI 状态 |
| `repositories` | 读写 IndexedDB、本地文件夹、WebDAV 等数据源 | 编排复杂业务流程 |
| `services` | 组合多个 repository 和 domain 规则完成业务用例 | 写复杂 UI 状态 |
| `stores` | 保存当前工作区、当前文档、加载状态等响应式状态 | 直接操作底层存储细节 |
| `features` | 页面、业务组件、交互流程 | 直接访问 IndexedDB/WebDAV/Worker 底层实现 |
| `shared` | 跨模块复用的 UI 和工具 | 引入具体业务模块形成反向依赖 |
| `workers` | 搜索建索引、关系计算等耗时任务 | 直接操作 Vue 状态 |

## 5. 核心领域模型

### 5.1 Document

```ts
export type DocumentId = string

export interface KnowledgeDocument {
  id: DocumentId
  type: 'document'
  title: string
  content: string
  tags: string[]
  parentId: DocumentId | null
  summary: string
  createdAt: string
  updatedAt: string
  isPinned: boolean
  isFavorited: boolean
  isPreset: boolean
  source: DocumentSource
}

export interface KnowledgeFolder {
  id: DocumentId
  type: 'folder'
  title: string
  parentId: DocumentId | null
  createdAt: string
  updatedAt: string
  isPinned: boolean
}

export type KnowledgeNode = KnowledgeDocument | KnowledgeFolder

export type DocumentSource =
  | { kind: 'indexeddb' }
  | { kind: 'local-fs'; path: string }
  | { kind: 'preset'; path: string }
```

### 5.2 Workspace

```ts
export interface Workspace {
  id: string
  name: string
  mode: 'indexeddb' | 'local-fs'
  status: 'ready' | 'loading' | 'permission-required' | 'error'
}
```

### 5.3 AI

```ts
export interface AiMessage {
  role: 'system' | 'user' | 'assistant'
  content: string
}

export interface AiProvider {
  id: string
  name: string
  type: 'online' | 'local' | 'ollama'
  listModels(): Promise<AiModel[]>
  chat(options: AiChatOptions): Promise<AiChatResult>
}
```

## 6. 关键模块设计

### 6.1 文档模块

文档模块是系统核心，其他功能都应该依赖它的稳定接口。

目标接口：

```ts
documentService.list()
documentService.get(id)
documentService.create(input)
documentService.update(id, patch)
documentService.remove(ids)
documentService.move(id, parentId)
documentService.togglePin(id)
documentService.toggleFavorite(id)
documentService.import(json)
documentService.export(ids)
```

注意事项：

- 文档树由 service 或 selector 计算，不在页面中散落实现。
- 文件夹和文档统一为 `KnowledgeNode`，避免 `isFolder` 到处判断。
- 所有写操作由服务层负责更新索引、刷新缓存、触发事件。

### 6.2 存储模块

用统一 repository 接口屏蔽不同存储来源。

```ts
export interface DocumentRepository {
  list(): Promise<KnowledgeNode[]>
  get(id: string): Promise<KnowledgeNode | null>
  save(node: KnowledgeNode): Promise<KnowledgeNode>
  delete(id: string): Promise<void>
  batchDelete(ids: string[]): Promise<void>
}
```

实现：

- `IndexedDbDocumentRepository`
- `LocalFsDocumentRepository`
- `PresetDocumentRepository`

WebDAV 不直接作为实时 repository，而作为同步/备份服务更合适。

### 6.3 Markdown 模块

Markdown 能力应该统一服务化，供编辑器、阅读器、静态文档、AI 归档共同使用。

目标能力：

- Markdown 渲染 HTML。
- 提取标题目录。
- 提取纯文本摘要。
- 提取 `[[WikiLink]]`。
- 同步任务清单复选框状态。
- 渲染 Mermaid。
- 渲染 Excalidraw。
- 解析本地图片链接并懒加载。

### 6.4 搜索模块

搜索应从 Store 中抽离，独立维护索引生命周期。

目标能力：

- 初始化索引。
- 增量添加、更新、删除文档。
- 后台 Worker 搜索。
- 返回带高亮片段的搜索结果。
- 支持标题、正文、标签权重。

```ts
searchService.indexAll(documents)
searchService.upsert(document)
searchService.remove(id)
searchService.search(query, options)
```

### 6.5 AI 模块

AI 模块要区分 Provider、会话和具体功能。

Provider：

- Online OpenAI-compatible Provider
- Local WebLLM Provider
- Local Transformers Provider
- Ollama Provider

业务功能：

- Chat：多轮对话。
- Write：根据标题生成文档。
- Polish：润色选中文本。
- Summary：总结文档。
- MindMap：生成导图大纲。

所有 AI 请求必须支持：

- `AbortController`
- 流式输出
- 错误归一化
- 模型配置
- 请求超时

### 6.6 图谱模块

图谱不应该直接扫描页面状态，而应该依赖 `relationService`。

关系来源：

- 文档父子关系。
- Markdown 双向链接。
- 标签共现。
- 收藏、置顶等辅助状态。

目标输出：

```ts
relationService.buildDocumentGraph(documents)
relationService.buildTagGraph(documents)
relationService.extractWikiLinks(document)
```

### 6.7 同步模块

WebDAV 同步建议作为独立服务，不和普通保存流程强绑定。

目标能力：

- 手动同步。
- 打开时同步。
- 定期备份。
- 同步日志。
- 冲突检测。
- 最小可用策略：本地导出 JSON 到 WebDAV。
- 进阶策略：按文档增量同步和冲突合并。

## 7. 迁移策略

推荐采用“渐进式重构”，避免一次性推倒重写。

### 7.1 阶段策略

1. 先搭建新目录和 TypeScript 能力。
2. 抽离领域模型，不改变现有页面。
3. 抽离 IndexedDB repository，保持原 UI 不变。
4. 抽离 `documentService`，让旧 Store 改为调用 service。
5. 重构编辑器保存链路。
6. 重构阅读器和 Markdown 服务。
7. 重构搜索服务。
8. 重构 AI 服务和聊天。
9. 最后整理图谱、思维导图、设置、同步和首页统计。

### 7.2 兼容策略

- 保留旧数据结构读取能力。
- 启动时将旧结构规范化为新结构。
- 写入时统一使用新结构。
- 对用户数据做迁移前备份。
- 每个阶段都保证应用可运行。

## 8. 验收标准

### 8.1 核心功能验收

- 可以加载旧 IndexedDB 文档。
- 可以新建、编辑、保存、删除文档。
- 可以创建文件夹和移动文档。
- 可以搜索标题、正文和标签。
- 可以正常渲染 Markdown、代码块、任务清单、Mermaid、数学公式。
- 可以使用 AI 对话和编辑器 AI 功能。
- 可以导出和导入数据。

### 8.2 架构验收

- 页面组件不直接调用 IndexedDB、本地文件夹、WebDAV。
- Store 不再包含复杂存储逻辑和搜索索引重建逻辑。
- 文档相关操作集中在 `documentService`。
- Markdown 解析集中在 `markdownService`。
- AI Provider 有统一接口。
- 新增功能可以放入对应 feature，不需要修改多个无关模块。

### 8.3 质量验收

- `npm run build` 通过。
- 核心服务有基础单元测试。
- 至少覆盖文档 CRUD、搜索、Markdown 双链解析、数据迁移。
- 手工回归清单通过。

## 9. 风险与处理

| 风险 | 表现 | 处理方式 |
| --- | --- | --- |
| 重构范围过大 | 长时间无法交付 | 按阶段迁移，每阶段可运行 |
| 用户数据损坏 | 文档读取失败或丢失 | 迁移前自动导出备份，保留旧结构兼容 |
| 编辑器行为回归 | 自动保存、图片、Markdown 互转异常 | 编辑器单独作为高风险模块验收 |
| 本地文件夹权限问题 | 刷新后需要重新授权 | Workspace 状态显式化 |
| AI 兼容性差 | 不同 Provider 行为不一致 | 统一 Provider 接口和错误模型 |
| 搜索卡顿 | 大量文档索引阻塞 UI | Worker 化和增量索引 |

## 10. 第一版 V2 范围

V2 第一版建议只追求稳定核心体验：

- 文档树
- 文档 CRUD
- Markdown 编辑与阅读
- 自动保存
- IndexedDB 存储
- 标签
- 搜索
- 数据导入导出
- 基础设置

后置功能：

- WebDAV 高级冲突合并
- 本地 AI 体验优化
- 复杂图谱交互
- 思维导图高级样式
- SEO 自动生成
- 首页全部统计图表
