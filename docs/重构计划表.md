# 知识库项目重构计划表

## 1. 总体节奏

本计划采用渐进式重构，不建议一次性推倒重写。每个阶段都应保持项目可运行、可回退、可验证。

预计周期：4 到 7 周。

优先级说明：

- P0：必须先做，影响整体架构和数据安全。
- P1：核心体验相关，影响主要使用路径。
- P2：增强体验，允许后置。
- P3：优化和收尾。

## 3. 详细任务拆分

### 阶段 0：功能冻结与盘点

目标：避免边重构边加功能，先确定要迁移什么。

任务：

- 列出当前所有页面和功能点。
- 标记每个功能的状态：保留、调整、删除、后置。
- 标记高风险功能：编辑器、数据存储、图片、AI、同步。
- 确定 V2 第一版 MVP 范围。
- 生成手工回归清单。

交付物：

- `docs/功能盘点.md`
- `docs/重构方案.md`
- `docs/测试验收清单.md`

### 阶段 1：架构骨架设计

目标：搭建新目录结构，让新代码可以逐步迁移进来。

任务：

- 新增并优先落地 `src/domain`、`src/repositories`、`src/services`。
- `src/app`、`src/features`、`src/shared` 作为目标结构保留，后续按页面迁移，不强制一次性搬目录。
- 明确模块依赖方向。
- 路由和布局暂保留原目录，等页面重构时再迁入 `src/app` 或 `features/layout`。
- 新增架构说明文档。

验收：

- 项目仍能启动。
- 不影响现有页面。
- 新旧目录可以共存。

### 阶段 2：TypeScript 基础接入

目标：先让核心层能使用 TS，不强制全部页面一次迁移。

任务：

- 新增或完善 `tsconfig.json`。
- 允许 `.ts` 与 `.js` 混用。
- 为路径别名 `@` 做类型配置。
- 新增核心类型文件。
- 先迁移纯工具模块，不动复杂 Vue 页面。

验收：

- `npm run build` 可通过。
- JS 文件无需立即全量改为 TS。

### 阶段 3：文档领域模型

目标：统一系统内最关键的数据结构。

任务：

- 定义 `KnowledgeDocument`、`KnowledgeFolder`、`KnowledgeNode`。
- 统一 `id`、`parentId`、`createdAt`、`updatedAt`、`tags`、`source` 字段。
- 编写旧数据到新数据的 normalize 函数。
- 编写文档树构建函数。
- 编写移动节点的合法性校验函数。

验收：

- 旧文档可以转换为新模型。
- 文件夹不能被移动到自己的子孙节点。
- 文档树排序规则清晰。

### 阶段 4：存储层抽象

目标：从业务逻辑中抽离底层存储。

任务：

- 定义 `DocumentRepository` 接口。
- 实现 `IndexedDbDocumentRepository`。
- 实现 `LocalFsDocumentRepository` 的最小能力。
- 将预设文档读取做成独立加载器。
- WebDAV 暂不作为实时 repository，先作为同步服务。

验收：

- IndexedDB 模式文档可读写。
- 本地文件夹模式可列出和保存 Markdown。
- 上层不关心当前用的是哪种 repository。

### 阶段 5：文档服务层

目标：让文档业务从 Store 中迁出。

任务：

- 新增 `documentService`。
- 实现 list/get/create/update/remove/move。
- 实现 togglePin/toggleFavorite。
- 实现 import/export。
- Store 改为调用 service，并保存加载状态和当前文档。
- 文档写入成功后触发搜索索引更新事件。

验收：

- 原有文档树、创建、编辑、删除、移动、置顶、收藏可用。
- Store 代码明显变薄。

### 阶段 6：编辑器链路重构

目标：把当前编辑器大组件拆成可维护模块。

建议拆分：

```text
features/editor/
├── EditorPage.vue
├── components/
│   ├── EditorToolbar.vue
│   ├── EditorTags.vue
│   ├── EditorToc.vue
│   └── EditorStatusBar.vue
├── composables/
│   ├── useEditorDocument.ts
│   ├── useAutoSave.ts
│   ├── useEditorPreview.ts
│   ├── useEditorImages.ts
│   └── useEditorAi.ts
└── editorExtensions.ts
```

任务：

- 将自动保存抽成 `useAutoSave`。
- 将文档加载保存抽成 `useEditorDocument`。
- 将 Markdown 预览抽成 `useEditorPreview`。
- 将图片保存和懒加载抽成 `useEditorImages`。
- 将 AI 帮写、润色、总结抽成 `useEditorAi`。
- 将 Tiptap extensions 配置单独维护。

验收：

- 编辑、新建、保存、自动保存正常。
- 分屏预览和阅读进度正常。
- 图片粘贴保存正常。
- AI 帮写和停止生成正常。

### 阶段 7：Markdown 服务统一

目标：避免编辑器预览、阅读器、静态文档重复处理 Markdown。

任务：

- 抽离 `markdownService.render`。
- 抽离 `extractHeadings`。
- 抽离 `extractWikiLinks`。
- 抽离 `generateSummary`。
- 抽离任务清单 checkbox 同步。
- 统一 Mermaid、轻量公式、代码复制、Excalidraw 渲染入口。

验收：

- Viewer、Editor 预览、MdDocs 使用同一套 Markdown 服务。
- 双向链接点击行为一致。
- 静态文档和动态文档样式一致。

### 阶段 8：搜索服务重构

目标：搜索不再依赖 Store 内部实现。

任务：

- 定义搜索文档 DTO。
- 将 FlexSearch 初始化迁入 `searchService`。
- 搜索 Worker 化。
- 支持增量 upsert/remove。
- 返回高亮摘要。
- 为搜索页和全局搜索提供统一接口。

验收：

- 文档保存后搜索结果自动更新。
- 删除文档后不会搜到旧文档。
- 大量文档下输入搜索不明显卡顿。

### 阶段 9：AI 模块重构

目标：统一不同 AI 引擎，降低调用方复杂度。

任务：

- 定义 `AiProvider` 接口。
- 实现 `OnlineAiProvider`。
- 实现 `OllamaAiProvider`。
- 实现 `LocalWebLlmProvider` 和 `LocalTransformersProvider`。
- 定义统一错误类型。
- 定义统一流式输出接口。
- 将 ChatView 迁移到 `chatService`。
- 将编辑器 AI 能力迁移到 `aiService` 上层用例。

验收：

- 在线 API、Ollama、本地模式配置可用。
- 对话可停止生成。
- 编辑器 AI 帮写、润色、总结可用。
- 错误提示可读。

### 阶段 10：关系图谱重构

目标：把图谱数据生成从页面中抽离。

任务：

- 新增 `relationService`。
- 从文档树生成父子关系。
- 从 Markdown 生成双链关系。
- 从标签生成共现关系。
- GraphView 只负责渲染图谱。

验收：

- 文档模式图谱可用。
- 标签模式图谱可用。
- 双向链接关系可用。

### 阶段 11：思维导图整理

目标：保持功能的同时降低单页复杂度。

任务：

- 整理导图数据模型。（已完成：节点模型、Markdown 解析、JSON 清理已迁入 `domain/mindmap`）
- 拆分持久化、布局、节点操作、导入导出。（已完成：持久化已抽到 `mindMapStorageService`，布局、节点操作、撤销重做、导入导出已抽到 `composables/mindmap`）
- AI 生成导图改用统一 AI 服务。（已完成：页面通过 `mindMapService` 生成 Markdown 大纲）
- 整理工具栏、侧栏、样式面板。
- 继续抽离页面内拖拽换父级、键盘快捷键、内联编辑和画布交互状态。

验收：

- 导图新增、编辑、删除、拖拽、撤销重做可用。（已具备）
- Markdown 导入可用。（已具备）
- JSON/PNG 导出可用。（已具备）
- AI 生成可用。（已具备）
- 页面剩余逻辑继续收敛到 composable 后，`MindMapView.vue` 只保留视图编排和事件绑定。

### 阶段 12：设置与同步整理

目标：设置页不直接塞满各种服务细节。

任务：

- 将设置持久化抽成 `settingsRepository`。
- 将 WebDAV 操作抽成 `syncService`。
- 本地工作区状态迁入 `workspaceStore`。
- 为测试连接、同步、备份提供统一任务状态。
- 记录同步日志。

验收：

- 设置修改可持久化。
- WebDAV 测试和同步可用。
- 本地工作区挂载状态清晰。

### 阶段 14：清理与发布

目标：移除旧代码和临时兼容层。

任务：

- 更新 README。

验收：

- `npm run build` 通过。
- 应用主要路径通过。

## 5. 里程碑

| 里程碑 | 内容 | 判断标准 |
| --- | --- | --- |
| M1：核心模型稳定 | 文档模型、存储接口、服务层完成 | 文档 CRUD 不依赖旧 Store 业务逻辑 |
| M2：核心体验可用 | 编辑器、阅读器、搜索完成 | 可以日常写文档和搜索 |
| M3：增强能力恢复 | AI、图谱、思维导图恢复 | 旧核心功能基本迁回 |
| M4：质量发布 | 测试、构建、文档、清理完成 | 可作为 V2 版本发布 |

## 6. 每阶段工作方式

每个阶段遵循同一流程：

1. 写清楚目标和影响范围。
2. 新增或迁移代码。
3. 保留旧入口，确保应用可运行。
4. 用小范围手工流程验证。
5. 通过后再进入下一阶段。
6. 最后统一删除旧代码。

## 7. 不建议做的事

- 不建议一开始就全量 TypeScript 改造。
- 不建议一开始就重写所有页面。
- 不建议先改 UI 再改数据层。
- 不建议在重构期间继续添加大功能。
- 不建议让 WebDAV 同步和本地保存强耦合。
- 不建议让 Tiptap JSON 成为核心业务数据格式，核心格式仍应是 Markdown。

## 8. 当前最优下一步

当前不再从阶段 0 重启。最优顺序是：

1. 收尾阶段 11：继续把 `MindMapView.vue` 中的拖拽、键盘、内联编辑和画布状态抽到 composable。
2. 推进阶段 6：从编辑器保存链路开始拆分，优先抽 `useEditorDocument`、`useAutoSave`、`useEditorPreview`。
3. 推进阶段 9：统一在线 API、Ollama、本地模型的 Provider 接口和错误模型，再迁移 ChatView 与编辑器 AI。
4. 推进阶段 12：把 WebDAV、本地工作区和设置持久化从页面/Store 中继续收敛到服务层。
5. 每完成一个阶段后执行 `docs/测试验收清单.md` 中对应模块的构建和手工回归项。
